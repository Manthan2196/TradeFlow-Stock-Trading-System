────────────────────────────────────────────────────────────────
TABLE 1 — users
────────────────────────────────────────────────────────────────
Purpose:
  Stores every person registered on the platform — investors,
  company representatives, and administrators. This is the
  central table that almost every other table connects back to.

Constraints:
  • PRIMARY KEY (user_id) — uniquely identifies each registered
    user across the entire platform.

  • UNIQUE (email) — prevents two accounts from sharing the
    same email address, since email is used for login.

  • NOT NULL (name, email, password, role, kyc_status,
    is_active) — every user must have these fields filled.
    A user without a name, email, or password cannot exist.

  • CHECK chk_role: role IN ('USER', 'ADMIN', 'COMPANY')
    — restricts role to only valid options. Prevents typos
    or invalid roles like 'SUPERUSER' from being inserted.

  • CHECK chk_kyc_status: kyc_status IN ('PENDING',
    'VERIFIED', 'REJECTED') — limits KYC status to known
    valid states only. Prevents garbage values.

  • DEFAULT role = 'USER' — new registrations automatically
    start with the lowest privilege level unless specified.

  • DEFAULT kyc_status = 'PENDING' — every new user starts
    as unverified and must be approved by admin.

  • DEFAULT is_active = TRUE — accounts are active by default
    and can be deactivated later without deleting them.


────────────────────────────────────────────────────────────────
TABLE 2 — wallet
────────────────────────────────────────────────────────────────
Purpose:
  Holds the money balance for each user. Every user has exactly
  one wallet which is used to fund stock purchases and receive
  proceeds from stock sales.

Constraints:
  • PRIMARY KEY (wallet_id) — uniquely identifies each wallet
    in the system.

  • UNIQUE (user_id) — ensures one user can only have one
    wallet. No user can hold two wallets.

  • FOREIGN KEY (user_id) REFERENCES users(user_id)
    ON DELETE CASCADE — links wallet to its owner. If the
    user account is deleted, the wallet is deleted too.

  • NOT NULL (user_id, balance, last_updated) — every wallet
    must belong to a user and must have a balance at all times.

  • CHECK chk_wallet_balance: balance >= 0 — a wallet balance
    can never go negative. Prevents users from spending
    money they do not have.

  • DEFAULT balance = 0 — every new wallet starts with zero
    balance until the user makes a deposit.

  • DEFAULT last_updated = NOW() — automatically records the
    creation time. Also updated by trigger on every change.


────────────────────────────────────────────────────────────────
TABLE 3 — wallet_transactions
────────────────────────────────────────────────────────────────
Purpose:
  Permanent log of every deposit (CREDIT) and withdrawal
  (DEBIT) made by users. Provides a full financial history
  for auditing and user transaction history pages.

Constraints:
  • PRIMARY KEY (transaction_id) — uniquely identifies each
    transaction record.

  • FOREIGN KEY (wallet_id) REFERENCES wallet(wallet_id)
    ON DELETE CASCADE — links each transaction to a wallet.
    If a wallet is deleted, its transaction history is
    also deleted.

  • NOT NULL (wallet_id, amount, transaction_type,
    created_at) — every transaction must have these values.
    An amount-less or type-less transaction is invalid.

  • CHECK chk_txn_type: transaction_type IN ('CREDIT',
    'DEBIT') — only two valid money movement directions
    are allowed. No other values accepted.

  • DEFAULT created_at = NOW() — transaction time is
    automatically captured the moment it is recorded.


────────────────────────────────────────────────────────────────
TABLE 4 — stocks
────────────────────────────────────────────────────────────────
Purpose:
  Master list of all stocks available for trading on the
  platform. Every stock has a ticker symbol, company name,
  and sector. Admins manage this list.

Constraints:
  • PRIMARY KEY (stock_id) — uniquely identifies each stock
    listed on the platform.

  • UNIQUE (symbol) — no two stocks can share the same ticker
    symbol (e.g., only one 'INFY' can exist).

  • NOT NULL (symbol, company_name, sector, is_active,
    created_at) — every stock must have a symbol, name,
    and sector. These are mandatory fields.

  • DEFAULT sector = 'General' — if no sector is provided
    when adding a stock, it defaults to 'General'.

  • DEFAULT is_active = TRUE — newly added stocks are
    available for trading immediately by default.

  • DEFAULT created_at = NOW() — records when the stock
    was added to the platform.


────────────────────────────────────────────────────────────────
TABLE 5 — stock_price_history
────────────────────────────────────────────────────────────────
Purpose:
  Stores every price recorded for every stock over time.
  Used to display price charts, calculate % change, and
  power the fn_top_gainers and fn_stock_price_change functions.

Constraints:
  • PRIMARY KEY (price_id) — uniquely identifies each price
    entry in the history.

  • FOREIGN KEY (stock_id) REFERENCES stocks(stock_id)
    ON DELETE CASCADE — links price record to its stock.
    If the stock is deleted, all its price history goes too.

  • NOT NULL (stock_id, price, price_timestamp) — every
    price record must know which stock it belongs to, what
    the price was, and when it was recorded.

  • CHECK chk_price_positive: price > 0 — a stock price
    of zero or below is not valid. Every recorded price
    must be a positive number.

  • DEFAULT price_timestamp = NOW() — automatically records
    the exact time when the price was logged.


────────────────────────────────────────────────────────────────
TABLE 6 — orders
────────────────────────────────────────────────────────────────
Purpose:
  Records every buy and sell order placed by users. An order
  starts as PENDING, gets EXECUTED when filled, or becomes
  CANCELLED if the user cancels it.

Constraints:
  • PRIMARY KEY (order_id) — uniquely identifies each order
    placed on the platform.

  • FOREIGN KEY (user_id) REFERENCES users(user_id)
    ON DELETE CASCADE — links order to the user who placed it.
    Deleting a user removes all their orders.

  • FOREIGN KEY (stock_id) REFERENCES stocks(stock_id)
    ON DELETE CASCADE — links order to the stock being traded.

  • NOT NULL (user_id, stock_id, order_type, order_price,
    quantity, order_status) — all key fields of an order are
    mandatory. An incomplete order cannot be inserted.

  • CHECK chk_order_type: order_type IN ('BUY', 'SELL')
    — only two types of orders are allowed. No other values.

  • CHECK chk_order_price: order_price > 0 — an order
    cannot be placed at a zero or negative price.

  • CHECK chk_order_quantity: quantity > 0 — must order at
    least 1 share. Zero or negative quantity is rejected.

  • CHECK chk_order_status: order_status IN ('PENDING',
    'EXECUTED', 'CANCELLED') — restricts status to valid
    lifecycle states only.

  • DEFAULT order_status = 'PENDING' — every new order
    starts as pending until it is processed or cancelled.

  • DEFAULT created_at = NOW() — order placement time is
    automatically recorded.


────────────────────────────────────────────────────────────────
TABLE 7 — trades
────────────────────────────────────────────────────────────────
Purpose:
  Records the actual execution of an order — the price and
  quantity at which shares were filled. One order can result
  in multiple trade records (partial fills).

Constraints:
  • PRIMARY KEY (trade_id) — uniquely identifies each trade
    execution record.

  • FOREIGN KEY (order_id) REFERENCES orders(order_id)
    ON DELETE CASCADE — links trade to the order it fulfilled.
    Deleting an order removes its trade records.

  • NOT NULL (order_id, executed_price, executed_quantity,
    trade_time) — all execution details are mandatory. A
    trade without a price or quantity is invalid.

  • DEFAULT trade_time = NOW() — exact execution time is
    automatically captured when the trade record is inserted.


────────────────────────────────────────────────────────────────
TABLE 8 — portfolio
────────────────────────────────────────────────────────────────
Purpose:
  Tracks what stocks each user currently holds and at what
  average price they bought them. This is the live inventory
  of a user's stock holdings used for P&L calculations.

Constraints:
  • PRIMARY KEY (portfolio_id) — uniquely identifies each
    holding row in the portfolio.

  • FOREIGN KEY (user_id) REFERENCES users(user_id)
    ON DELETE CASCADE — links holding to the owner.

  • FOREIGN KEY (stock_id) REFERENCES stocks(stock_id)
    ON DELETE CASCADE — links holding to the stock held.

  • UNIQUE (user_id, stock_id) — a user can have only ONE
    row per stock. No duplicate holdings for the same stock
    per user. Buying more shares updates the existing row.

  • NOT NULL (user_id, stock_id, total_quantity,
    avg_buy_price) — all fields are mandatory. A portfolio
    entry without a quantity or price is invalid.

  • CHECK chk_portfolio_qty: total_quantity >= 0 — a user
    cannot hold a negative number of shares.
    Also enforced at DB level by trigger trg_portfolio_positive.

  • DEFAULT total_quantity = 0 — starts at zero when the
    first order for a stock is pending.

  • DEFAULT avg_buy_price = 0 — starts at zero and updates
    as buy orders are executed.


────────────────────────────────────────────────────────────────
TABLE 9 — portfolio_transactions
────────────────────────────────────────────────────────────────
Purpose:
  Detailed log of every change in portfolio quantity — every
  buy that increased shares and every sell that decreased them.
  Provides a full history of how a portfolio evolved.

Constraints:
  • PRIMARY KEY (pt_id) — uniquely identifies each portfolio
    change record.

  • FOREIGN KEY (portfolio_id) REFERENCES portfolio(portfolio_id)
    ON DELETE CASCADE — links change record to the holding it
    affected. Deleting a portfolio entry removes its history.

  • FOREIGN KEY (order_id) REFERENCES orders(order_id)
    — optional link to the order that caused the change.

  • FOREIGN KEY (trade_id) REFERENCES trades(trade_id)
    — optional link to the trade execution that caused it.

  • NOT NULL (portfolio_id, quantity_change, created_at)
    — every record must know which portfolio changed and
    by how much.

  • DEFAULT created_at = NOW() — timestamp is automatically
    recorded at the moment of the change.


────────────────────────────────────────────────────────────────
TABLE 10 — company_profiles
────────────────────────────────────────────────────────────────
Purpose:
  Stores the official profile of each company listed on the
  platform. Links a COMPANY-role user to their stock ticker.
  One company profile = one user account = one stock.

Constraints:
  • PRIMARY KEY (company_id) — uniquely identifies each
    company profile.

  • UNIQUE (user_id) — a company user account can only be
    linked to one company. One user = one company.

  • UNIQUE (stock_id) — each stock can only be linked to one
    company. No two companies can own the same stock.

  • FOREIGN KEY (user_id) REFERENCES users(user_id)
    ON DELETE CASCADE — if the company user is deleted,
    their company profile is also deleted.

  • FOREIGN KEY (stock_id) REFERENCES stocks(stock_id)
    ON DELETE CASCADE — if the stock is removed, the
    company profile linked to it is also removed.

  • NOT NULL (user_id, stock_id, company_name, verified,
    created_at) — core identity fields are all mandatory.

  • DEFAULT verified = FALSE — a newly created company profile
    is unverified by default. Admin must approve it before
    the company gains full platform access.

  • DEFAULT created_at = NOW() — profile creation time
    is automatically recorded.


────────────────────────────────────────────────────────────────
TABLE 11 — company_announcements
────────────────────────────────────────────────────────────────
Purpose:
  Stores news and updates posted by companies — such as
  dividend declarations, quarterly results, AGM notices,
  and stock split information. Visible to investors.

Constraints:
  • PRIMARY KEY (announcement_id) — uniquely identifies each
    announcement.

  • FOREIGN KEY (company_id) REFERENCES company_profiles(company_id)
    ON DELETE CASCADE — links announcement to its company.
    If the company profile is deleted, all its announcements
    are deleted too.

  • NOT NULL (company_id, title, content, announcement_type,
    is_published) — every announcement must have a title,
    body text, type, and published status.

  • CHECK chk_ann_type: announcement_type IN ('DIVIDEND',
    'SPLIT', 'BONUS', 'RESULTS', 'AGM', 'OTHER') — only
    recognised announcement categories are accepted.

  • DEFAULT announcement_type = 'OTHER' — if no type is
    specified, it defaults to a general 'OTHER' category.

  • DEFAULT is_published = FALSE — announcements are saved
    as drafts by default. Company must manually publish them.

  • DEFAULT created_at = NOW() — creation timestamp is
    automatically set.

  • DEFAULT updated_at = NOW() — also auto-updated by
    trigger trg_ann_updated_at whenever the announcement
    is edited.


────────────────────────────────────────────────────────────────
TABLE 12 — company_financials
────────────────────────────────────────────────────────────────
Purpose:
  Stores quarterly financial results uploaded by companies —
  revenue, net profit, and earnings per share (EPS). Investors
  use this data to evaluate company performance.

Constraints:
  • PRIMARY KEY (financial_id) — uniquely identifies each
    financial report entry.

  • FOREIGN KEY (company_id) REFERENCES company_profiles(company_id)
    ON DELETE CASCADE — links financial record to its company.
    Deleting the company removes its financial data.

  • UNIQUE (company_id, quarter, fiscal_year) — a company can
    only upload one financial record per quarter per year.
    Prevents accidental duplicate submissions.

  • NOT NULL (company_id, quarter, fiscal_year, revenue,
    net_profit, eps, published_at) — every financial record
    must be complete. No partial records allowed.

  • CHECK chk_quarter: quarter IN ('Q1', 'Q2', 'Q3', 'Q4')
    — only four valid quarter values accepted.

  • DEFAULT published_at = NOW() — records when the
    financial data was submitted to the platform.

TABLE 13 — audit_logs
────────────────────────────────────────────────────────────────
Purpose:
  Permanent record of every significant action on the platform.
  Written automatically by triggers — not by application code.
  Used by admins to investigate activity and ensure
  accountability. Records are never deleted.

Constraints:
  • PRIMARY KEY (audit_id) — uniquely identifies each
    audit log entry.

  • FOREIGN KEY (user_id) REFERENCES users(user_id)
    ON DELETE SET NULL — links log to the user who acted.
    If user is deleted, the log entry is kept with
    user_id set to NULL (audit trail must be preserved).

  • NOT NULL (action, resource, status, created_at) — every
    log entry must record what happened, on which table,
    and whether it succeeded.

  • CHECK chk_audit_status: status IN ('SUCCESS', 'FAILED',
    'WARNING') — restricts outcome values to known states.

  • DEFAULT status = 'SUCCESS' — most actions succeed, so
    success is assumed unless otherwise specified.

  • DEFAULT created_at = NOW() — exact timestamp of the
    action is captured automatically.


────────────────────────────────────────────────────────────────
TABLE 14 — database_transactions
────────────────────────────────────────────────────────────────
Purpose:
  Tracks the lifecycle of database transactions — whether they
  are still active, successfully committed, or rolled back.
  Used by the trg_dbtxn_log trigger to write system logs.

Constraints:
  • PRIMARY KEY (txn_id) — uniquely identifies each
    database transaction record.

  • FOREIGN KEY (user_id) REFERENCES users(user_id)
    ON DELETE SET NULL — links transaction to the user
    who triggered it. Log is kept even if user is deleted.

  • NOT NULL (txn_state, start_time) — every transaction
    must have a state and a start time.

  • CHECK chk_txn_state: txn_state IN ('ACTIVE',
    'COMMITTED', 'ROLLED_BACK') — only three valid lifecycle
    states for a database transaction.

  • DEFAULT txn_state = 'ACTIVE' — every new transaction
    starts as active until it ends.

  • DEFAULT start_time = NOW() — start time is automatically
    captured when the transaction is opened.


────────────────────────────────────────────────────────────────
TABLE 15 — system_logs
────────────────────────────────────────────────────────────────
Purpose:
  Low-level system event records automatically written by the
  trg_dbtxn_log trigger when a transaction commits or rolls
  back. Stores duration, operation name, and outcome.

Constraints:
  • PRIMARY KEY (log_id) — uniquely identifies each system
    log entry.

  • FOREIGN KEY (txn_id) REFERENCES database_transactions(txn_id)
    ON DELETE SET NULL — links log entry to its transaction.
    Log is kept even if the transaction record is removed.

  • NOT NULL (operation, status, timestamp) — every system
    log must record what operation occurred and its result.

  • DEFAULT status = 'SUCCESS' — defaults to success unless
    the trigger records a failure.

  • DEFAULT timestamp = NOW() — log write time is
    automatically captured.


────────────────────────────────────────────────────────────────
TABLE 16 — locks
────────────────────────────────────────────────────────────────
Purpose:
  Records which database transactions are currently holding
  locks on specific resources. Used to track and manage
  concurrent access to critical data like wallets and orders.

Constraints:
  • PRIMARY KEY (lock_id) — uniquely identifies each lock
    record.

  • FOREIGN KEY (txn_id) REFERENCES database_transactions(txn_id)
    ON DELETE CASCADE — links lock to its transaction.
    When the transaction ends, all its locks are released
    and the lock records are deleted.

  • NOT NULL (txn_id, resource_type, lock_mode,
    acquired_at) — every lock must specify who holds it,
    what resource is locked, and what kind of lock it is.

  • CHECK chk_lock_mode: lock_mode IN ('SHARED',
    'EXCLUSIVE') — only two valid lock types. SHARED allows
    multiple readers; EXCLUSIVE blocks everyone else.

  • DEFAULT lock_mode = 'EXCLUSIVE' — write operations take
    an exclusive lock by default to prevent conflicts.

  • DEFAULT acquired_at = NOW() — lock acquisition time
    is automatically recorded.


────────────────────────────────────────────────────────────────
TABLE 17 — system_config
────────────────────────────────────────────────────────────────
Purpose:
  Key-value store for platform-wide settings managed by
  admins — such as trading limits, maintenance mode flags,
  or fee percentages. Acts as the platform's control panel.

Constraints:
  • PRIMARY KEY (config_id) — uniquely identifies each
    configuration entry.

  • UNIQUE (config_key) — no two settings can have the same
    key name. Prevents duplicate or conflicting config entries.

  • NOT NULL (config_key, config_value) — every setting must
    have both a key and a value. An empty setting is useless.

  • FOREIGN KEY (updated_by) REFERENCES users(user_id)
    ON DELETE SET NULL — records which admin last changed
    the setting. If admin user is deleted, the config entry
    is kept with updated_by set to NULL.

  • DEFAULT updated_at = NOW() — records when the setting
    was last modified.


================================================================
  CONSTRAINT TYPES — QUICK REFERENCE
================================================================

  PRIMARY KEY   → Uniquely identifies each row. No duplicates,
                  no nulls allowed. Every table has exactly one.

  FOREIGN KEY   → Ensures the referenced row actually exists
                  in the parent table. Prevents orphan records.

  ON DELETE CASCADE  → If parent row is deleted, all child
                       rows are automatically deleted too.

  ON DELETE SET NULL → If parent row is deleted, the FK column
                       in the child is set to NULL (row kept).

  UNIQUE        → No two rows can have the same value in this
                  column (or combination of columns).

  NOT NULL      → The column must always have a value.
                  Empty/missing values are rejected.

  CHECK         → The value must satisfy a condition.
                  Invalid values are rejected by the database.

  DEFAULT       → If no value is provided on INSERT, the
                  database automatically fills this in.

================================================================
END
